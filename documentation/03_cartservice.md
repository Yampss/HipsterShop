# Cart Service

## Overview
The Cart Service is responsible for managing the user's shopping session (adding, retrieving, and emptying items in the cart). We migrated this service from using an in-memory Redis cache to a durable MongoDB database.

* **Language:** C# (.NET Core)
* **Port Exposed:** 7070 (gRPC)

## How the Code Works
The core logic resides in `MongoCartStore.cs`. 
1. The service receives a `userId` (a UUID generated by the Frontend stored in a browser cookie).
2. For an `AddItem` request, it connects to MongoDB, reads the existing `cart_data` (which is a serialized Protobuf `BinData` blob stored in a BSON document), deserializes it into C# objects, increments the quantity, reserializes it, and saves it back using an upsert operation (`ReplaceOneAsync` with `IsUpsert = true`).
3. `Startup.cs` manages dependency injection, choosing to instantiate our `MongoCartStore` class because the `MONGO_CONNECTION_STRING` environment variable is detected at startup.

## Potential Trainer Questions

**Q: Wait, why did you store the cart data as a binary blob instead of a structured document?**
**A:** "This is a tactical design choice. Since the `CartItem` schema is defined strictly by our `demo.proto` protobuf file, we simply serialize the strongly-typed Protobuf object directly into bytes and store it as BSON `BinData`. This gives us the durability of MongoDB with replication, while keeping the lightning-fast read/write patterns and avoiding complex schema migrations when the protobuf definition changes."

**Q: How does this service securely connect to the database in Kubernetes?**
**A:** "We use Kubernetes `Secrets`. Inside our `cartservice.yaml` manifest, we use `valueFrom: secretKeyRef` to inject the MongoDB password into the container's environment variables at runtime. The full connection string with replica set is constructed from this password, ensuring no credentials are hardcoded in the C# source code."
